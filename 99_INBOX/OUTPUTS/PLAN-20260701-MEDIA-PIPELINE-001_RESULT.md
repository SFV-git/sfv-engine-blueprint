# Serial Queue Processing Rules  

## Why Serial Processing is Required  
Due to the limited VRAM capacity of the single GPU machine, **parallel processing** of video/photo files through FFmpeg (transcoding) and Whisper/faster-whisper (transcription) is **not feasible**. Both processes are memory-intensive:  
- **FFmpeg** requires significant VRAM for video decoding/encoding, especially for high-resolution media.  
- **Whisper/faster-whisper** consumes substantial VRAM during audio feature extraction and model inference.  
Running these tasks in parallel would exceed the GPU's memory limits, leading to crashes, out-of-memory errors, or unstable performance. **Serial processing** ensures deterministic resource allocation and avoids contention, guaranteeing completion of each job without failure.  

## Queue Priority Order  
Jobs in the queue are processed in the following priority order to optimize client experience and system efficiency:  
1. **Client-facing urgent tasks** (e.g., real-time transcription requests, user-initiated transcoding with deadlines).  
2. **Batch processing jobs** (e.g., bulk media conversions, scheduled transcription tasks).  
3. **Low-priority background tasks** (e.g., metadata extraction, thumbnails generation).  
Prioritization is enforced by the queue manager, which assigns jobs to the processing pipeline based on their defined urgency and resource requirements.  

## Handling Stuck/Hung Jobs  
To prevent indefinite blocking of the pipeline:  
- **Timeout Mechanism**: Each job is monitored for a predefined timeout (e.g., 15 minutes). If a job exceeds this duration without completing, it is flagged as **stuck**.  
- **Skip Logic**: Upon timeout, the system logs the failure, skips the job, and proceeds to the next item in the queue. Temporary files generated by the job are automatically cleaned up to free resources.  
- **Root Cause Analysis**: Administrators are notified of timeouts for investigation, with logs including GPU memory usage, job status, and stack traces if applicable.  

## Retry Policy  
A simple retry policy is applied to transient failures:  
- **Maximum Retries**: 3 attempts per job.  
- **Retry Delays**: Exponential backoff (e.g., 1 minute, 2 minutes, 4 minutes between retries).  
- **Conditions for Retry**: Retries are triggered only for **transient errors** (e.g., temporary GPU memory spikes, I/O errors). **Permanent failures** (e.g., invalid input files, unsupported codecs) are skipped immediately.  
- **Dead-letter Queue**: Jobs that fail after 3 retries are moved to a dead-letter queue for manual inspection and resolution.  

This approach balances reliability, resource constraints, and user expectations.
