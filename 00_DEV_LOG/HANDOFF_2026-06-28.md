### THE BIG ONE — root cause of all cloud-model failures
- Symptom: `hermes -z "PING"` → "API call failed after 3 retries: Connection error." Telegram replies came back as the masked error template "model provider failed after retries."
- Chased it down: NOT a model-name issue, NOT paid-tier (though we confirmed free Portal tier = `paid_access:false`, so **Claude via Portal needs a paid plan** — Claude on Portal is NOT free).
- **TRUE ROOT CAUSE:** `httpx` raised `[SSL: CERTIFICATE_VERIFY_FAILED] unable to get local issuer certificate`. TLS probe showed the cert issuer = **"Norton Web/Mail Shield Root … generated by Norton Antivirus for SSL/TLS scanning."** Norton is MITM-ing all HTTPS; Windows trusts its root, Python's certifi bundle does not. Breaks Portal, direct Anthropic, AND was the same cause as the original eval's uv "UnknownIssuer" install errors. Local Ollama was unaffected (plain http).
- **FIX APPLIED:** `pip install truststore` in BOTH venvs (`app\.venv` and `hermes-agent\venv`) + wrote `sitecustomize.py` into both `site-packages` dirs that calls `truststore.inject_into_ssl()` on every Python start (makes Python use the Windows cert store, which trusts Norton).
  - Direct verify PASSED: `truststore.inject_into_ssl()` then httpx GET to Portal `/v1/models` → `TLS_OK 200`.
  - Files: `...\app\.venv\Lib\site-packages\sitecustomize.py` and `...\hermes-agent\venv\Lib\site-packages\sitecustomize.py`.
