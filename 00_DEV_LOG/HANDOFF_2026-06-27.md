### What was researched this session
- Claude Code headless mode (-p flag): fully documented, production-ready. Fires from scripts/cron/n8n.
  Accepts prompt, runs agent loop (plan+execute), exits. --max-turns and --max-budget-usd cap runaway.
  --output-format json makes output parseable for orchestration.
- Hermes Agent (Nous Research, MIT license): released Feb 2026, 200k+ GitHub stars.
  Persistent daemon, cross-session memory, self-writes skill docs from experience (closed learning loop),
  cron scheduling, 18+ messaging platforms (Telegram/Discord/Slack etc.), 7 terminal backends,
  MCP support, works with Anthropic API + Ollama + any provider.
  Most relevant: spawns Claude Code sessions for coding tasks, returns results async.
  Self-improvement: after 20+ self-created skills, completes similar tasks 40% faster (token/time).
  Desktop app shipped June 2, 2026 (v0.15.2). MIT license, free. Pay only API costs.
  Repo: github.com/NousResearch/hermes-agent

### The flaw in Will's original loop design
Will proposed: Opus (planner) → Ollama (executor) → Opus (pinged) → Antigravity review every 3-5 steps.
Three structural problems:
1. Ollama (qwen3:14b) cannot execute engine-build work — it classifies/summarizes. Claude Code builds.
2. Claude Chat cannot be "pinged" by a file — it has no daemon. Only Claude Code headless can be triggered.
3. Multi-agent hand-off chains degrade past 2 hand-offs (persona drift, state loss) — confirmed by
   PromptQuorum May 2026 benchmark of 6 real stacks. Every extra hop is a failure point.

### Corrected architecture (FOR HUMAN REVIEW — needs Will ratification)

The roles:
- Claude Code headless (claude -p): the PLANNER + EXECUTOR in one seat. Reads directive, builds, commits, writes result. This is the thing that actually builds the engine.
- n8n + localFileTrigger (already working): the TRIGGER. Sees result file → fires next claude -p.
- Ollama (qwen3:14b, cheap, always-on): the GATEKEEPER. Before each step fires, classifies: in-scope or human-gate? Routes to HANDOFFS/ if human input needed. Does NOT build.
- Claude Code headless (review-only prompt, read-only tools): INNER REVIEWER every 3-5 steps. Checks drift vs directive, writes REVIEW_NNN.md. No Antigravity rate limit consumed.
- Antigravity: FINAL REVIEWER only. Fires once per completed directive phase. Rare, rate-limit-safe.
- Will: writes CURRENT_DIRECTIVE.md once per phase (30-60 min). Reviews HANDOFFS/. Approves phases.

The loop:
  Will writes CURRENT_DIRECTIVE.md
    → n8n fires first claude -p (reads directive + SESSION_STATE + CRITICAL_PATH)
    → Claude Code plans + executes one step → writes to STEP_RESULTS/
    → Ollama gate: in-scope? → n8n fires next claude -p | human-gate? → HANDOFFS/
    → every 3-5 steps: claude -p review pass → REVIEW_NNN.md
    → directive complete → Antigravity full review → happy: done | unhappy: new directive
    → Will sees result in messaging app (Hermes delivers) or HANDOFFS/

### Build vs adopt decision (FOR HUMAN REVIEW)

OPTION A — ADOPT Hermes Agent as the persistent daemon + orchestrator
  Hermes handles: always-on daemon, cron, messaging delivery, skill memory, Claude Code spawning.
  We build: SFV-specific skills, the CURRENT_DIRECTIVE pattern, Ollama gate node in n8n, directive templates.
  Pro: 200k stars, actively maintained, MIT, already does 80% of the loop.

  n8n triggers it. Existing infrastructure only.
  Pro: no new dependency, fully custom, sits on confirmed working stack.
  Con: build time, maintenance burden, reinventing what Hermes already does.
  Estimated build: 1 session for Claude Code to scaffold.
